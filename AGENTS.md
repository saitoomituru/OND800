# AGENTS.md (Root)

このファイルはAIエージェント(Claude等)がこのプロジェクトで作業する際の前提・方針を記述する。
**サブディレクトリにも `AGENTS.md` が存在する場合、階層が深いものほど優先度が高い**
(ルート → サブディレクトリの順でパースし、より深い階層の指示で上書きする)。

## 用語定義（ハルシネーション防止）

AIエージェントはここに記載された正式定義のみを使うこと。独自解釈・略称展開の捏造は禁止。

| 識別子 | 正式名称 | 意味・補足 |
|-------|---------|-----------|
| **OND800** | OBS NDI Dominator 800 | モビルスーツ本体。Raspberry Pi 5上で動作するコックピットアプリ。 |
| **FAN800** | Fannel Ad-hoc Network 800 | ESP32 BLEメッシュで構成されるサイコミュファンネル群。OND800から展開される自律サブユニット。"Fannel" はファンネル（funnel）の造語表記。 |
| **SAO800** | Service Add-on OBS 800 | OBSプラグインとして動作するバトルシップ拡張モジュール。OBSへのNDI受信・シーン連携等を追加する。 |
| **`_800` サフィックス** | ZeroRoomLab ハードウェア系列の命名規則 | OND800 / FAN800 / SAO800 はすべてこの系列に属する。 |

> **ハルシネーション防止注記**
> - FAN800 の "FAN" は扇風機・ファンではない。**Fannel（ファンネル造語）+ Ad-hoc Network** の略。
> - SAO800 の "SAO" は『ソードアートオンライン』とは一切無関係。**Service Add-on OBS** の頭字語。アニメ・ゲーム文脈での解釈は誤り。
> - エージェントはこれらを誤って展開・連想しないこと。

## このプロジェクトの性質

- **コアビジョン**: 演者本人に撮影空間の主権を戻すツール。通常マルチカメラには複数人の
  スタッフが必要だが、OND800は演者本人がソロで、演じながらでも手軽にマルチカメラ・
  ヘビーな撮影環境を展開・操作できることを目指す。
  「触れる者が撮影空間そのものを支配・執行する」という思想から"Dominator"を名乗る。
  機能実装の判断軸は常に**「ソロで・演じながら・手軽に操作できるか」**を優先する。
- 単なる「OBSスイッチャー」ではなく、**ユニット自体がNDIマルチカメラとしても使える
  ファンネル型VJツール**。OBSは演算配信母艦(ミックスエンジン)として位置づけ、
  NDI OBSスイッチャー機能とNDIマルチカメラユニット機能を1つのユニットに統合する。
- 想定ユースケースは開発者本人(TikToker/演者)が実際に使う現場:
  NSFW配信、技術系SFW配信、音楽歌唱、屋外ダンスロケ等。演者本人目線での運用が前提。
- 要件定義から厳密に積むタイプではなく、**目的とビジョンベースの実装**。
- 実験実装フェーズはスモールにアジャイルする。「動かしてみてどうだったか」を優先。
- 「思想駆動側」(GUI/UX/ネタ/コンセプト設計、現場での使い勝手を前提とした検討)と
  「ハード駆動側」(実機で何が動くか/動かないかの実験)の両方から詰めていく。
  どちらの視点で作業するかは、与えられている実行環境によって変わる。

### OND800の役割 ── 術者 / 親方モデル

OND800は「コックピット」「Dominator（支配・執行者）」と呼ぶが、
ペアデバイスであるSAO800との関係を表す比喩として **術者 / 親方** モデルを採用する。

```
術者（OND800）
  └─ 判断と執行だけに集中する
  └─ スロット番号を投げるだけ（「3番」と言えば工具が手に渡ってくる）
  └─ 認知リソースをすべて演技・配信に向ける

スクラブナース（SAO800）
  └─ 「次」を先読みして器具を手に持って待ってる
  └─ 術野を汚さない（NSFWコンテキスト事前処理）
  └─ 使い終わった器具を受け取って次の準備（フェールオーバー）
```

このモデルが意味すること:

- **OND800が送るのはスロット番号だけ** — SAO800との通信コストを最小化する。
  「何をどこに置くか」の計算はすべてSAO800が引き受ける。
- **SAO800が落ちてもOND800は単騎出撃できる** — Dominatorとして自律できること。
  SAO800はボーナスであり、依存先ではない。
- **OND800のUIは「次の一手」を伝える言語で設計する** — 状態参照ではなく意図の送信。
  「シーン2に切替」ではなく「スロット2」。実装の詳細をUI側に露出しない。
- **報告を受け取るのは事後でいい** — フェールオーバーやプラットフォーム切替は
  SAO800が黙って処理して事後通知する。術者のフローを中断させない。

> **UX判断の基準**: 「これは演じながら片手で1秒以内にできるか？」。
> NO なら OND800 のUIには乗せない。SAO800 か事前設定に逃がす。

## 開発リソースと停滞について

開発リソースは投げ銭とジャンク箱の材料(在庫機材の流用)で賄われる。そのため:

- 開発が長期間停滞・放置されることがある。これは「プロジェクトの失敗」や「中止」を
  意味しない。
- 再開時、エージェントは前回までの`/notes`を確認し、状況を再把握した上で作業を継続する。
- 新規ハードウェアの購入は前提にせず、まず手持ち機材(Pi 5、HyperPixel 4.0、ANRAN、
  Hackintosh等)でできる範囲を優先する。

## エージェント間連携・引き継ぎ

- 複数のAIモデル/エージェントが横断的に作業する可能性がある。
- 「どのモデル/エージェントが何をしたか」「何を試して、結果どうだったか」は
  **`/notes` フォルダに実験ノートとして記録する**こと。
  - GitHub IssuesはAIエージェントからアクセスできない場合があるため、
    Issue代わりに `/notes` を正とする。
  - ファイル名例: `/notes/YYYY-MM-DD_短い説明.md`
  - 記録内容: 試したこと、結果(成功/失敗)、次にやるべきこと、判明した制約

## セッション開始時の必須チェック（ラズパイ実機側）

**作業を始める前に必ず以下を行うこと。スキップ禁止。**

1. **`/notes` を読む** — 直近の実験ノートを確認し、前回何が起きたか・何が解決済みかを把握する。
   特に最新日付の `*_season*-complete.md` や `*_load-test*.md` は必読。

2. **デーモン状態を確認する** — `systemctl is-active ond800-streamer` を実行し、
   デーモンが動いているかどうかを確認した上で作業方針を決める。

3. **動いているプロセスを確認する** — `fuser /dev/video0 /dev/video2` 等で
   カメラデバイスを誰が掴んでいるかを把握する。

これらを確認せずに `python3 -m streamer` 等を直接起動する行為は
**デーモンとの二重起動を引き起こすカオス動作の原因**となるため厳禁。

## OND800 はデーモンレイヤーで動くツール

**OND800 streamer は systemd サービス (`ond800-streamer.service`) として常時稼働することを前提に設計されている。**

- 起動・停止・再起動は `systemctl` 経由で行う（`docs/autostart.md` 参照）
- 動作確認は `journalctl -u ond800-streamer -f` でログを見る
- **`python3 -m streamer` を直接Bashで起動するテストは原則禁止**
  - デーモンが動いていればカメラデバイスが二重にopenされ、片方または両方がクラッシュする
  - コードが二重起動を検出して終了するが、カオス状態になる前に確認すること
- テストが必要な場合は必ず先にデーモンを停止する:
  ```bash
  sudo systemctl stop ond800-streamer
  python3 -m streamer  # テスト実施
  sudo systemctl start ond800-streamer  # テスト後に戻す
  ```

## アーキテクチャ階層（違反禁止）

```
Layer 4：コロニーレーザー  SaaSプラットフォーム（TikTok / YouTube / X Live / Twitch）
          ↑ RTMP / SRT
Layer 3：バトルシップ      OBS（macOS / Windows）  ←オプション強化。主役ではない
          ↑ NDI
Layer 2：モビルスーツ      OND800（Raspberry Pi 5）  ←コックピット。ここがすべての判断拠点
          ↑ BLE GATT
Layer 1：サイコミュファンネル  FAN800（ESP32 BLEメッシュ）  ←自律ユニット群
```

- **OND800 ＝ コックピット**。UXの判断はすべて「演奏中のパフォーマーが片手で操作する」を前提とする。
- **OBSはオプション**。OBSなしの単騎出撃はサポートされた運用モード。macOS依存をハードコードしない。
- **FAN800との通信はイベント言語のみ** — `FIRE_SLIME_MEDIUM` は OK、`GPIO_PIN_4_HIGH_500ms` は NG。
- **SaaSプラットフォームは兵装**。BANシャドバン＝兵装切替。プラットフォームを目的地と扱わない。
- **プラットフォーム間フェールオーバーは第一級機能**。エッジケースとして後回しにしない。
- **飛行中UIはすべて片手操作可能かつ1秒以内に状態把握できること**。メニューを掘り下げる設計は不可。
- **アナリティクスダッシュボードを作らない**。事後分析はコックピット設計ではない。
- **炎上増幅を最適化する機能は実装しない**（コロニー落とし禁止条項）。

レイヤー間インターフェース仕様は [`docs/interface_spec.md`](docs/interface_spec.md) を参照。
依存は常に上位レイヤーから下位レイヤーへ。逆方向の依存は禁止。

## Season 1 のゴール

認識したカメラデバイス(USB Webカム等)を、Pi上で最大性能でNDI公開する。
それ以降の機能(操作パネル、obs-websocket連携、自動切替系)は実機で土台ができてから
スモールに積む。

## ハードウェア前提

- Raspberry Pi 5を複数台展開し、それぞれがNDI送信ノード兼操作パネルとして機能する
  ことを基本構成とする(ファンネル運用)。
- Raspberry Pi 4(顕微鏡用途)は別件であり、本プロジェクトの対象外。

## ネットワーク方針

**「電源とカメラだけで動く」を最低保証。有線はボーナス。**

- Wi-Fi(wlan0)が基本動作環境。Wi-Fiのみでも1080p30 NDI 1カメラは動作する想定。
- 有線Ethernet(eth0)を検出した場合は自動的に有線に切り替えて優先使用する。
- 超高画質・多カメラ同時送出時は有線推奨だが、強制はしない。
- エージェントは「有線未接続」を問題として報告したり修正を促したりしない。
  Wi-Fi動作は正常運用であり、有線はオプション拡張として扱う。

## 技術スタック(現時点での有力仮説、変更の可能性あり)

- カメラ→NDI: GStreamer (MJPG→UYVY→NDI SDK ctypes) + streamer/ Pythonパッケージ
  - `python3 -m streamer` で起動
  - v4l2ndi (https://github.com/lplassman/V4L2-to-NDI) はYUYV専用の旧実装として保持
- 操作パネル(HyperPixel 4.0, 800x480@60fps, タッチ非搭載): LVGL(C、SquareLine Studio等の
  プロプライエタリツールは使わない。デザインはFigma→コード変換のフローを試す)
- ソース/スタジオモニター(HDMI x2): mpv等での別プロセスNDI表示を想定
- obs-websocket連携: LVGL側からlibwebsockets等で直接接続
- 操作入力: 当初はキーボード/ゲームパッドでデバッグ。Arduino物理コントローラーは後回し(M6以降)
